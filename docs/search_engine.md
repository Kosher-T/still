# The Hybrid Search Engine

This document specifies exactly how Still finds Bible verses during a live sermon. The search engine operates in five sequential phases, executing in approximately 40–80 milliseconds per cycle.

---

## Phase 1: The Sliding Window Handoff

**Time elapsed: 0 ms**

Still uses a Continuous Stream (CS) STT model. The search engine does not wait for acoustic pauses — it is driven entirely by a fixed word-count buffer.

### Buffer Mechanics

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| **Search Window size** | 15 words | Provides sufficient semantic context for `all-MiniLM-L6-v2` to generate meaningful 384-dimensional embeddings. The proposed 10-word/4-overlap reduction was rejected to preserve vector integrity. |
| **Trigger Buffer** | 50 words | A secondary rolling RAM array decoupled from the search window. Allows directional look-back without destroying mathematical vector integrity. |
| **Trailing overlap** | Last 6 words retained | Bridges context across consecutive search iterations; guarantees intent trigger phrases up to 7 words are caught. |
| **Trigger mechanism** | Strictly word-count based | Acoustic pause triggers are explicitly omitted to conserve GPU compute and reduce system complexity. |

### Execution & Decoupled Buffering

1. Thread 2 (STT Inference) appends transcribed words to the primary 15-word buffer, while simultaneously maintaining the 50-word rolling **Trigger Buffer**.
2. If an intent trigger is detected:
   - **Preceding Search:** Extract the prior 15 words from the 50-word Trigger Buffer instantly and push to Queue B.
   - **Proceeding/Centered Search:** Set a "Wait State" flag. Halt that specific search execution temporarily, wait for the incoming STT stream to provide the necessary future words, then snap the 15-word window and fire it.
3. Under normal continuous speech without a trigger, when the primary buffer hits exactly 15 words, the full text block is pushed to Queue B.
4. The primary buffer drops the oldest 9 words and retains the last 6 as the trailing overlap for the next iteration.

### Slow Speech — The Deadlock Threshold & TTL Override

If the pastor speaks fewer than 15 words and then pauses (e.g., 14 words followed by a 2-minute musical transition), the buffer has a strict **Time-To-Live (TTL) timeout** of 3–5 seconds of silence.

The TTL Override is an **absolute global interrupt**. If an intent trigger invokes a "Wait State", but the pastor pauses longer than the TTL, the system:
1. **Forcefully flushes** the partial buffer.
2. Executes a partial search on whatever words are in the buffer.
3. Applies **Dynamic RRF Scaling** (see Phase 3 below).
4. **Clears the Wait State**. It never waits indefinitely. Retaining stale text across a long pause would mathematically destroy the geometric vector coordinates generated by the semantic model.

---

## Phase 1.5: The 8-Word BM25 Early Exit

To catch spontaneous, unannounced scripture with zero latency, an 8-word micro-trigger bypasses the 15-word wait.

**Operator Setting (`require_trigger_for_fast_lane`):**
- **False (Default):** BM25 executes continuously every 8 words.
- **True (Strict Mode):** The 8-word check is entirely suppressed unless a regex intent trigger evaluates to True.

## Phase 2: Parallel Processing — The Two Lanes

The 15-word text block is duplicated and fired down two parallel search lanes simultaneously.

### Lane A: BM25 Lexical Search (Exact Keyword Matching)

**Time to complete: ~5 ms**

BM25 does not understand meaning — it matches exact words against the 186,000+ verses in the offline Bible database.

#### Step 1: Lexical Normalization & 6-Version Matrix

**The 6-Version Matrix:** The FAISS and BM25 databases are rigidly locked to exactly six versions to prevent RRF signal dilution and RAM bloat: **KJV, NKJV, ESV, NIV, NLT, AMP** (Amplified Bible).

Faster-Whisper outputs capitalized, punctuated text (e.g., "The", "love."), which breaks exact BM25 matches. Before stop-words are stripped, the incoming string is explicitly normalized via `str.translate()`:
- Apostrophes (`'`) are stripped to nothing ("don't" -> "dont").
- Compound/boundary punctuation (`-`, `—`, `/`, `:`) are replaced with spaces ("first-born" -> "first born", "John 3:16" -> "John 3 16").
- The string is forcefully converted to lowercase.

> [!CAUTION]
> **Symmetric Mapping:** This exact mapping is enforced symmetrically during offline index building and live search execution to guarantee Whisper output matches the BM25 tokens perfectly.

*This mutation occurs strictly inside Lane A.* Lane B (FAISS) receives the raw capitalized and punctuated text because the embedding model relies on exact punctuation to map syntactic relationships properly.

#### Step 2: Stop-Word Stripping

To guarantee **absolute deterministic stripping** and avoid regression risks from third-party library updates (NLTK, spaCy, etc.), Still uses a **custom, hardcoded array** converted into a hash set in memory. The list is deliberately minimal:

```python
STOP_WORDS = {"the", "is", "a", "and", "to", "of", "in", "that"}
```

> [!IMPORTANT]
> **Archaic and Biblical vocabulary is never stripped.** Words like "thou," "hath," and "unto" are intentionally retained as high-value lexical fingerprints. This is the core of the "Fingerprint Strategy" — because the semantic lane (FAISS) already handles meaning and paraphrase matching, BM25's sole purpose is exact-text detection. Retaining archaic vocabulary allows BM25 to instantly distinguish between a KJV quote and a NIV paraphrase without any additional translation-checking logic.

#### Step 2: Scanning

BM25 scans the Bible database using term-frequency / inverse-document-frequency (TF-IDF) scoring to find the verses with the highest concentration of remaining keywords.

#### Step 3: Result

Returns a **Top 5** ranked list of verse candidates based on exact vocabulary overlap.

### Lane B: FAISS Semantic Search (Meaning and Paraphrase Matching)

**Time to complete: ~20–30 ms**

This lane uses the `all-MiniLM-L6-v2` embedding model (running via ONNX Runtime on CPU) and the in-memory FAISS vector index.

#### Step 1: Vectorization

The embedding model reads the 15-word string and converts it into a 384-dimensional mathematical coordinate (a dense vector).

#### Step 2: Distance Calculation

During initialization, every verse in the Bible database was pre-vectorized into the same 384-dimensional space and loaded into FAISS in RAM. FAISS computes the geometric distance (cosine similarity) between the new spoken-word vector and all 186,000+ verse vectors.

#### Step 3: Result

Returns a **Top 5** ranked list of verse candidates based entirely on semantic similarity — perfectly catching paraphrases, reworded quotes, and approximations.

## Phase 3: Fusion and Scoring

**Time elapsed: ~35 ms**

The two Top 5 lists must be merged into a single ranked result with a continuous 0–100% confidence score.

### Reciprocal Rank Fusion (RRF)

The standard RRF formula with a smoothing constant of **k = 60**:

$$RRF(d) = \frac{1}{60 + rank_{BM25}(d)} + \frac{1}{60 + rank_{Vector}(d)}$$

**Edge cases for single-lane appearances:**

| Scenario | Calculation |
|----------|-------------|
| Verse ranked #1 in **both** lanes | `1/(60+1) + 1/(60+1) ≈ 0.0327` (maximum possible) |
| Verse ranked #1 in BM25, absent from FAISS | `1/(60+1) + 0 ≈ 0.0164` |
| Verse ranked #5 in FAISS only | `0 + 1/(60+5) ≈ 0.0153` (minimum possible) |
| Verse ranked #3 in BM25, #2 in FAISS | `1/(60+3) + 1/(60+2) ≈ 0.0320` |

When a verse appears in only one lane, the missing lane contributes **zero** to the RRF score. The verse is still a valid candidate — it simply scores lower than a verse appearing in both lanes.

### Min-Max Normalization to 0–100%

The RRF output is converted to a human-readable confidence percentage:

$$Confidence = \left( \frac{RRF_{observed} - RRF_{min}}{RRF_{max} - RRF_{min}} \right) \times 100$$

| Variable | Value | Source |
|----------|-------|--------|
| $RRF_{max}$ | ≈ 0.0327 | Ranked #1 in both lanes |
| $RRF_{min}$ | ≈ 0.0153 | Ranked #5 in one lane only |

**Tuning:** If empirical testing shows that verses hovering at 60% are triggering false positives in the auto-display tier, the `RRF_min` variable is adjusted upward to compress the scale, pushing lower-ranked combinations toward lower percentages.

### Dynamic RRF Scaling (Partial Buffers)

When the buffer flushes early due to the slow-speech TTL timeout (e.g., only 8 words instead of 15), the semantic model inherently produces weaker vector coordinates due to insufficient context. To prevent unfairly penalizing short phrases:

- As the buffer word count drops below 15, the system **programmatically lowers `RRF_max`** in the normalization equation.
- This compresses the grading scale, allowing weaker raw RRF scores from a short phrase to still achieve the 85% auto-display threshold when the match is genuinely strong.

## Phase 4: Intent Classification

**Time elapsed: ~55 ms**

See [intent_classification.md](intent_classification.md) for the full specification. The search engine receives the intent evaluation as a boolean state:

| Trigger State | Meaning |
|-------------|---------|
| **True** | Explicit positive biblical trigger phrase identified (e.g., "Turn to") |
| **False** | Ambiguous or casual mention; defaults to False unless negative override found |

---

## Phase 5: The Display Decision

**Time elapsed: ~75 ms**

The final routing decision combines the Confidence Score and the Boolean Trigger State:

| Confidence | Trigger State | Action |
|-----------|---------------|--------|
| ≥ 85% | True | **Auto-Display** — bypass operator, push directly to WebSocket → OBS |
| ≥ 85% | False | **Operator Review Queue** — high confidence, but needs human validation |
| 40–84% | Any | **Operator Review Queue** — subject to LRU deduplication |
| < 40% | Any | **Discard** — text appended to transcript, search results dropped |

> [!NOTE]
> The exact confidence thresholds (85%, 40%) are **configurable by the system admin** from the operator settings menu. These values represent sensible defaults and should be adjusted based on empirical testing during rehearsals.

### LRU Cache Deduplication

Because the 6-word trailing overlap can cause the exact same verse to be identified in consecutive search iterations, a **Least Recently Used (LRU) cache** intercepts results before they reach the operator queue.

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| **Cache scope** | Verse reference string (e.g., "John 3:16") | Simple, deterministic key |
| **TTL** | 15 seconds | Long enough to suppress overlap duplicates; short enough that a deliberate re-quote 3 minutes later is treated as a fresh event |
| **Storage** | Standard RAM dictionary | Negligible memory footprint |

**Execution flow:**

1. The display decision determines a verse should go to the operator review queue.
2. The verse reference is checked against the LRU cache.
3. **Cache hit** → result is silently discarded as a duplicate.
4. **Cache miss** → reference is added to the cache with a 15-second TTL, and the verse is pushed to the operator's review queue.

> [!NOTE]
> Deduplication cannot happen at the search layer (Phase 2/3) because `all-MiniLM-L6-v2` requires the full 15-word context — including overlapping words — to generate accurate embeddings. Stripping overlaps before vectorization would degrade semantic accuracy.
